When building large web pages with a lot of content, it is possible to see a "bug" from Flotr2. The "bug" is intentional by design from the Flotr2 team. In the process of constructing charts Flotr2 creates dummy HTML elements so that it can calculate their sizes. Flotr2 does not intend the dummy elements to be visible on the page, so it "hides" them by positioning them off screen. What Flotr2 this is off the screen isn't always the case however.

Specifically line 2,281 of flotr2.js is:
D.setStyles(div, {"position": "absolute", "top": "-10000px" })

Flotr2 intends to place the dummy elements 10,000 pixels above the top of the browser window, however, CSS absolute positioning can be relative to the containing element which is not always the browser window. This can cause Flotr2 to scatter text in random-looking locations throughout the page. There are a few workarounds for this:

  • One option is to modify the open source code directly within the project, however, you would have to import the library to the project locally, you can not use the CDN.

  • Another option would be to close all the elements after the Flotr.draw() has been called for the last time.

    To close the elements, the following line can be used:
    document.getElementById("ELEMENT_ID_NAME").classList.toggle("show");

The workhorse of charts. Effective at showing the change of a quantity of over a small number of regular time intervals, or at comparing several different quantities against one another.

For this chart example, I'll use fictitious sales data for a bakery that's tracking bagels sold over the last 7 years.

As seen in the example to the right, there are three (3) layers of arrays.

  • Each data point consists of an x-value and a y-value packaged in an array
  • Each series consists of a set of data points packaged in an array
  • The data to chart consists of one or more series packaged in an array

Some questions that are still left unanswered from the previous example charts are:

  1. What countries are shown?
  2. Are there any regional differences?
  3. What's that datapoint all the way to the top right?

Let's answer those questions

To start, one way to answer those questions would be to add mouseovers (or tool-tips) to each data point. I will not use that approach in this example for a few reasons: First, I will be exploring interactive charts later. Secondly, mouseovers and tooltips are ineffective for users accessing a site on a touch device, such as a smartphone or tablet.

If it were a requirement that users have to mouse to fully understand a visualization, then they might be neglecting a significant (and rapidly growing) number of them.

My approach to this problem will be to divide the data into mutliple series so that I may appropriately shade and label each datapoint independently. There are changes to the code and data structure which have been updated in this section's code and data tabs.

Adds a third value to scatter charts but should be used carefully, as it's difficult to accurately assess the relative areas of circular regions.

If we need to visualize three (3) variables, we could use a scatter plot framework for two of the variables and then vary the size of the points according to the third variable. This results in a bubble chart.

Using bubble charts effectively requires some caution. Humans are not very good at accurately judging the relative areas of non-rectangular shapes, so bubble charts don't lend themselves to precise comparisons of the bubble size. If the third variable conveys only the general sense of a quantity rather than an accurate measurement, a bubble chart can be appropriate.

If there is data with many dimensions, a radar chart may be the most effective way to visualize it. Radar charts are not as common as other charts, though, and their unfamiliarity makes them a little harder for users to interpret. When designing radar charts, try to be cognizant to not increase the burden.

Radar charts are most effective when data has several characteristics:

  • When there are not too many data points to show. Half a dozen is about the maximum that a radar chart can accommodate.
  • The data points have multiple dimensions. With two or even three dimensions to the data would probably be better off with a more traditional chart type. Radar charts come into play with data of four or more dimensions.
  • Each data dimension is a scale that can at least be ranked (from good to bad), if not assigned a number outright. Radar charts do not work well with data dimensions that are merely arbitrary categories (such as political parties or nationality).

A classic usage of radar charts is analyzing the performance of players on sports teams.
